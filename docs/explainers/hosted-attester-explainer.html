The Hosted Attester

Give every agent a verifiable identity — without giving it a secret.

Software agents come and go by the thousand. You can't pre-register each one and hand it a password. The hosted attester lets a workload prove what it is and walk away with a key-bound credential — minted on demand, signed by a key that never leaves the vault, and usable only by the instance that asked for it.

The big picture

From source code to a bearer token — in one line.

Everything that follows is a detail of one journey. A workload starts life as ordinary code and ends holding a bearer access token for the API. The attester adds a single step in the middle — the one in gold — that turns a client registered on paper into a live, key-bound credential the running instance can actually wield.

The problem

A secret you can hand out is a secret you can lose.

The classic model — register a client, issue it a credential, trust it forever — falls apart the moment your "clients" are a fleet of ephemeral, autonomous agents. Three things break at once.

Scale

You don't onboard one client, you deploy a fleet that scales up and down by the minute. Minting and rotating a static secret per instance simply doesn't hold.

Provenance

A verifier needs to know what kind of agent is calling and where it runs — not just that it holds the right bytes. A leaked secret carries none of that.

Revocation

When a workload type is compromised or retired, you must deny it everywhere, at once — not chase down long-lived tokens scattered across an estate.

How it works

The workload proves its identity. PingFederate mints the credential.

The agent holds only its own ephemeral key and an instance identity from its runtime. It never signs the attestation, and never touches the attester's key.

present

Prove

The workload presents its instance attestation — a SPIFFE SVID, or a wallet's provider‑signed WIA — and a proof it holds its own instance key.

check

Verify

The attester validates that attestation against its trust root and confirms the instance is bound to this client.

bind

Mint

It binds the workload's key as cnf, folds the identity and entitlements in, and stamps an expiry you choose.

seal

Sign

It signs with the client's attester key — held in an OpenBao vault, never exposed to the runtime.

→The agent now authenticates with a credential bound to a key only it holds — nothing to replay, nothing to steal, and worthless to anyone who lifts a copy.

At issuance time · two layers of trust

The instance proves what it is. The client's rules come from one of three places.

Signing an attestation for an instance takes trust at two levels — kept separate on purpose. The instance layer is a pluggable format, not a fourth row in the table below; it's the layer those three client sources sit on top of. The credential is minted only when both line up.

The instance

What is this instance — in whatever form?

It presents an instance attestation: an infrastructure workload a SPIFFE JWT‑SVID; a digital wallet a Wallet Instance Attestation from its provider. A pluggable format, always vouched by its own trust root — the SPIFFE trust domain, the wallet provider. Always required, whichever client source is in play.

then →
The client

Whose instance is it — and what may it be granted?

The attester resolves the client the instance belongs to: its accepted instance trust roots, its registered instances, its entitlements. That's what these three sources establish.

The client layer — three sources, one credential out

Client sourceHow trust is establishedBest forAssurance
Local config Provisioned directly on the authorization server. Clients you operate yourself. operator
OpenID Federation A signed entity statement, cryptographically chained to a trust anchor — with live membership and instant revocation. Cross-organization ecosystems. high
CIMD The client's ID is a URL you fetch for its metadata — lightweight, no pre-registration. Open, self-service onboarding. self-hosted

Only when the two layers meet — the instance's SVID validates against the client's bundle, and its SPIFFE ID is a registered instance of that client — does the attester bind the instance's own key and sign the credential.

A deliberate safeguard: because a self-hosted (CIMD) document is unsigned, the attester never trusts a bundle it asserts about itself — the trust roots stay under the attester's control. High-assurance federation earns the right to publish its own.

Why it matters: a workload can't credibly self-assert its region, its tier, or its spending limit — those come from the attester's registration data, sealed into a credential bound to a key only this instance holds. The request says who; the attester decides what.

Why it's safer

Nothing worth stealing sits still.

no long-lived secret

The workload holds only its own key

The agent never possesses the attester's signing key, and its instance key is generated locally and never leaves it. There is no shared secret to exfiltrate.

keys stay in the vault

The signing key never enters the app

Attestations are signed inside an OpenBao transit engine with a least-privilege token — the private key never touches the authorization server's memory.

lifetime on your terms

As short or as long as the workload needs

Match the lifetime to the instance — tight for ephemeral jobs, so revocation is automatic the moment re-issuance stops; longer for durable, long-running agents. Whichever you pick, it's bound to a key only that instance holds, so a lifted copy is inert.

least privilege

Per-instance authority

Each workload identity carries only the fine-grained entitlements it needs (RFC 9396) — one client, many bound identities, each scoped independently.

Standards, end to end

No bespoke crypto. No proprietary lock-in.

Every hop is an open standard — the attestation, the key binding, the entitlements, the trust spine, and the identities themselves.
