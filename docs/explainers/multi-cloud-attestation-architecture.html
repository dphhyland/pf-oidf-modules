Enterprise architecture · multi-cloud workload identity

Every cloud runs its own SPIFFE. One federation makes them trust each other.

Each platform — EKS, AKS, GKE, bare metal — issues its workloads a SPIFFE identity from its own SPIRE trust domain. PingFederate runs inside each cloud: workloads authenticate to its attestation endpoint with their SVID, and it uses the OpenID Federation trust anchor to verify the SPIFFE-ID is registered to a client — the SPIFFE bundle and binding live in that client's federation entity record — before issuing a key-bound client attestation. One federation, honoured across every cloud.

Cross-cloud, end to end

A workload in AWS authenticates to a service in Google Cloud.

The same journey the diagram implies, followed one hop at a time — the caller never leaves its own cloud's identity, yet the credential travels the whole estate.

Get a local identity

The payments-agent pod asks its node's SPIRE agent over the Workload API and receives a JWT-SVID. No secret is provisioned to it, ever.

AWS · EKS · spiffe://aws.acme.example

Authenticate in-cloud

It presents the SVID to PingFederate running inside AWS — the attestation endpoint sits alongside SPIRE. SPIFFE is how the workload authenticates; there is no secret to hold.

AWS · PingFederate attestation endpoint

Verify registration, then issue

PingFederate resolves the client's federation entity record — the SPIFFE bundle and the spiffe_id ↔ client_id binding — validates the SVID against it, mints a key-bound attestation and issues an access token.

PingFederate ↔ OpenID Federation

Cross the boundary

It calls statements-api running in GKE. That service trusts the same federation, so the token is accepted — a different cloud, a different SPIRE, one trust fabric.

Google Cloud · GKE
Why federation is the spine

Keep each cloud's trust domain isolated. Stitch them with a chain, not a secret.

independent by design

One SPIRE trust domain per platform

Each cloud runs its own SPIRE with its own trust domain and keys. Nothing is shared across the boundary — a compromise in one cloud doesn't hand an attacker identity in another.

registered once

The registration lives in the entity record

A client's federation entity record carries its SPIFFE trust bundle and its spiffe_id ↔ client_id bindings in metadata. Any cloud's PingFederate resolves that record via the trust chain — so the bundle it validates the SVID against is federation-vouched, never self-asserted by the workload.

live control

Membership and revocation are central

Onboard a new platform by adding a subordinate to the federation; retire or revoke one the same way. Trust decisions are governed at the anchor, not scattered across clusters.

no secret sprawl

Nothing long-lived crosses the boundary

Workloads hold only their own ephemeral keys; the attestation is bound to that key and short by policy. There is no cross-cloud shared secret to distribute, rotate, or leak.

Standards, end to end

Every layer is an open standard.

Platform identity, the trust spine, the credential, and its key binding — nothing bespoke, nothing locked in.
